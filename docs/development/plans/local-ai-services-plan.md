# 本地 AI 服务规划（计划）

## 1. 文档定位

本文用于记录后续本地 STT 与本地翻译能力的结构规划。

本文是计划，不是当前实现说明；暂不锁死：

- 最终 provider 选型
- 具体目录名、文件名、配置键名
- 自动拉起、打包与分发形式

---

## 2. 规划目标

- 在补充本地能力时，不破坏主程序现有边界
- 在同时支持本地 STT 与本地翻译时，避免结构失控
- 让后续本地能力扩展复用同一套思路

---

## 3. 规划范围

- 本地 STT
- 本地翻译
- 本地模型依赖管理
- 本地模型缓存与运行时资源管理

---

## 4. 计划要点

### 4.1 主程序职责

主程序继续聚焦于：

- 音频采集
- 运行时控制与生命周期
- 统一事件流与后处理
- 文本稳定化
- VRChat 输出

主程序默认不承担本地模型推理的复杂职责。

### 4.2 本地能力接入方式

- 本地 STT 与本地翻译优先按独立本地服务或 sidecar 方向规划
- 主程序通过清晰边界接入本地能力
- 上层主链路不直接依赖某个本地模型的内部实现细节

### 4.3 依赖管理

- 主项目依赖保持精简
- 本地重依赖按对应服务分别管理
- 不将所有本地推理依赖直接并入主程序环境

### 4.4 模型与运行时资源

- 模型文件视为运行时资产，不进入源码仓库
- 模型缓存、运行日志、临时状态应分开管理
- 模型下载、升级与清理策略后续单独收敛

### 4.5 配置方向

- 主程序配置保持高层抽象
- 主程序只描述能力选择、服务连接与策略选择
- 模型、设备、缓存、推理细节优先放在对应本地服务侧

### 4.6 统一结构原则

本地 STT 与本地翻译尽量遵循同类规则：

- 都通过清晰边界接入主程序
- 都将重依赖隔离在核心运行时之外
- 都保留替换底层实现的空间
- 都避免把 provider 特有细节扩散到上层

---

## 5. 与里程碑的关系

- MVP 仍然坚持 CLI first、cloud STT first、original-language captions first
- 本规划主要服务于后续本地 STT 与翻译阶段
- 本地能力可先以 experimental 形态进入验证

---

## 6. 决策原则

后续为本地 AI 能力做结构决策时，优先选择：

- 更容易观察与诊断的方案
- 更容易隔离问题的方案
- 更容易替换底层实现的方案
- 对主运行时侵入更小的方案
- 更符合 Windows 主平台维护成本的方案

---

## 7. 当前落地说明（M5 第一版）

- 本地 STT 已按同包 sidecar 方向落地，而不是新建独立 `services` 项目。
- 主程序新增 `funasr_local` provider，通过本地 websocket 连接 `vrc-live-caption local-stt serve`。
- 主程序配置只保留 sidecar 连接信息。
- FunASR 模型、chunk、VAD、标点和线程等重参数收敛到 `local-stt-funasr.toml`。
- 当前默认设备策略为 `device = "auto"`：在 Windows + NVIDIA + `funasr-cu128` 环境下优先使用 `cuda:0`，否则回退到 CPU。
- 显式 `device = "cuda"` 采用快速失败策略，不做静默 CPU fallback。
- 当前 GPU 支持范围仅收敛到 Windows + NVIDIA + cu128；自动拉起、打包分发和模型自动下载仍不在本阶段范围内。
