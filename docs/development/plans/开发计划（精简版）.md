# 开发计划（精简版）

## 1. 文档定位

本文用于定义项目目标、MVP 边界、关键约束、阶段里程碑与主要风险。

本文不锁死以下内容：
- 具体目录结构
- 具体类名与接口命名
- 具体默认参数
- 本地 provider 的最终选择

凡是数值阈值、provider 排名、发送长度上限、刷新频率等实现细节，均视为初始建议或待验证项，最终以 M0 实测和后续迭代结果为准。

---

## 2. 版本定义

### 2.1 MVP
MVP 的目标是先跑通并验证这条最小主链路：

```
麦克风 -> 音频预处理 -> STT -> 文本稳定化 -> OSC -> VRChat Chatbox
```

MVP 优先级：
- 先 CLI，后 GUI
- 先云端高精度，后本地模式
- 先把原文字幕链路做稳，再补翻译能力

---

## 3. 项目目标

- 中文识别优先
- 中英夹杂场景可用
- 字幕滚动自然，不明显抖动或刷屏
- Windows 作为主要使用与验收平台
- macOS 仅作兼容性考虑，不是主要目标
- 架构上预留云端与本地双模式能力

---

## 4. MVP 范围

### 4.1 MVP 包含
- 麦克风音频采集
- 基础音频预处理与队列管理
- 云端流式 STT 接入
- partial / final 的统一事件流
- stable / unstable 的文本稳定化
- 通过 OSC 将字幕发送到 VRChat Chatbox
- 基础配置加载
- 日志、错误提示与基础诊断能力
- 基础 CLI 运行能力

### 4.2 MVP 不包含
- GUI
- 本地 STT 正式模式
- 实时翻译正式模式
- 自动 provider fallback
- 账号系统与云同步
- Web / 移动端
- 多人协作服务
- 高级插件系统

### 4.3 v1 计划补齐
- 本地 STT 模式
- 翻译层
- 基础 GUI
- Windows 打包与测试版交付能力

---

## 5. 核心原则与约束

### 5.1 架构原则
- 音频采集、STT、翻译、OSC 之间保持清晰分层
- provider 必须先归一化为统一事件，再进入后处理链
- GUI 只作为现有运行时与配置层的包装，不重写核心业务逻辑
- 不为了 v1 以外的功能提前引入过重的抽象

### 5.2 运行原则
- 音频回调线程只做轻量工作，不做重计算
- 队列必须有上限，出现积压时允许降级处理，但不能拖死主流程
- 网络请求、模型推理、翻译处理不得阻塞主运行流程
- 停止时必须能优雅退出，并尽量完成必要的 flush

### 5.2.1 VRChat Chatbox 已确认硬约束
- Chatbox 文本单次上限为 `144` 字符
- Chatbox 最多显示 `9` 行，自动换行与手动换行都计入上限
- `/chatbox/input s b n` 中 `b=true` 表示立即发送，`b=false` 仅写入输入框
- `/chatbox/input s b n` 中 `n=false` 可以关闭这次消息提示音
- `/chatbox/typing b` 用于切换 typing 指示

### 5.3 错误与降级原则
- 默认不做隐式自动切换 provider
- 云端失败时应明确重试、报错或退出，不做不可见的魔法行为
- 本地 provider 若能力不足，可以以降级或 experimental 形态提供
- OSC 发送异常不能拖垮主链路，但必须留下明确日志

### 5.4 隐私与密钥原则
- 默认不保存原始音频
- 默认不保存完整转写内容
- 调试录音或 trace 只在显式开启时启用
- API Key 不写入普通配置文件
- 日志默认避免输出完整用户语音内容

### 5.5 质量目标
- 字幕应具备可接受的实时性，云端模式优先达到可用水平
- partial 更新应及时，但不能造成明显刷屏
- 连续运行至少 30 分钟不崩溃、不出现明显资源失控
- 出现常见错误时，用户能够根据日志和诊断结果定位问题

---

## 6. 关键待验证项

除 `5.2.1` 中已确认的 VRChat Chatbox 硬约束外，这些问题不应在文档里被提前写死，而应在 M0 或后续阶段通过实测确认：

- VRChat 对刷新频率、typing 状态、清空行为的实际容忍度
- 中英文混排在 Chatbox 中的实际显示效果
- 云端 STT provider 的中文与中英夹杂表现
- 本地候选 provider 的中文效果、延迟与资源占用
- partial 抖动在真实语音中的表现，以及稳定化策略是否足够

---

## 7. 里程碑

### M0：可行性验证与规格收敛
目标：
- 把影响路线选择的高不确定性先验证掉

重点工作：
- 实测 VRChat Chatbox 行为边界
- 验证云端 STT 主方案可行性
- 准备固定测试语料
- 对本地候选 provider 做基础对比
- 收敛需要写进规格的关键默认值与限制

完成标志：
- 明确 Chatbox 约束
- 明确云端方案可行
- 明确本地 provider 的后续路线
- 明确哪些参数属于“默认值”，哪些属于“硬约束”

### M1：工程骨架与音频链路
目标：
- 搭出稳定的最小运行时基础

重点工作：
- 项目骨架
- 配置加载
- 日志与基础状态管理
- 音频设备枚举与输入采集
- 队列与基础预处理
- 调试录样能力

完成标志：
- CLI 可启动
- 可稳定采集麦克风音频
- 可优雅退出
- 配置和环境问题能给出明确提示

### M2：云端 STT 打通
目标：
- 先让高质量转写主链路跑通

重点工作：
- 接入云端流式 STT
- 统一 partial / final 事件
- 处理断线、超时、重试与失败退出
- 建立 provider 状态观测

完成标志：
- CLI 能持续输出实时转写结果
- 中文与中英夹杂场景基本可用
- 常见网络异常有明确行为

### M3：字幕稳定化与 VRChat 输出
目标：
- 让结果真正可用于 VRChat 中的实时字幕体验

重点工作：
- stable / unstable 稳定化
- revision 去重与节流
- 提交与 flush 规则
- OSC 文本发送
- 文本裁剪、去重与发送节奏控制
- 基础诊断命令与关键日志补齐

完成标志：
- VRChat 中能看到连续滚动字幕
- partial 不会明显刷屏
- final 能稳定提交
- 具备 MVP 所需的基础诊断能力

### M4：MVP 验收与稳定性加固
目标：
- 把主链路从“能跑”推进到“可连续使用”

重点工作：
- 连续运行测试
- 资源占用观察
- 常见错误场景回归
- 配置、日志、诊断信息打磨

完成标志：
- 满足 MVP 验收标准
- 可以作为首个外部测试版本使用

### M5：本地 STT 模式
目标：
- 在不破坏主链路结构的前提下补齐离线能力

重点工作：
- 确定本地 provider 方案
- 模型缓存、缺失提示、初始化失败处理
- 适配 provider 能力差异带来的降级行为

完成标志：
- 无网时可运行基础原文字幕链路
- 本地模式至少可作为 experimental 提供

### M6：翻译层
目标：
- 在不阻塞主链路的前提下补齐跨语言字幕能力

重点工作：
- 翻译层解耦
- 先做云端 text-to-text 翻译，不接 LLM API
- 首版仅支持 final_only
- partial 原文实时显示，final 译文异步覆盖
- stale 结果丢弃与翻译队列有界
- 原文 / 译文 / 双语输出模式
- 双语模式按 `source\ntarget` 渲染

完成标志：
- 翻译延迟不会拖垮主链路
- 输出模式切换清晰
- 双语模式在 VRChat `144` 字符与 `9` 行约束下可用
- 翻译能力可作为 v1 特性进入测试

### M7：基础 GUI 与打包
目标：
- 将 CLI 原型包装成基础桌面测试版

重点工作：
- 启停控制
- provider 与音频设备选择
- 基础设置项
- 实时预览与日志查看
- Windows 打包

完成标志：
- 不使用命令行也能完成基本操作
- GUI 不破坏既有核心架构
- Windows 可交付测试版

---

## 8. MVP 验收标准

MVP 通过的最低标准：

- 用户说话后，VRChat Chatbox 中能看到连续滚动的原文字幕
- 中文场景明显可用
- 中英夹杂场景不至于明显失效
- 云端模式下整体延迟处于可接受范围
- 连续运行 30 分钟不崩溃
- 常见问题能够通过日志和诊断定位

说明：
- MVP 不要求 GUI
- MVP 不要求本地 STT 成熟可用
- MVP 不要求正式翻译能力
- MVP 不要求自动 fallback

---

## 9. 主要风险

### 风险 1：VRChat Chatbox 的行为限制比预期更强
应对：
- 提前实测
- 将发送策略独立封装
- 允许根据实测结果调整文本裁剪与发送节奏

### 风险 2：partial 抖动过大，影响可读性
应对：
- 稳定化逻辑前置为核心能力
- 做 revision 去重与节流
- 保留以 final 为准的提交机制

### 风险 3：本地中文 STT 效果不稳定
应对：
- 本地模式后置
- 允许 experimental 形态发布
- 先保证云端体验

### 风险 4：音频链路或队列积压导致卡顿
应对：
- 音频回调只做轻量工作
- 队列有界
- 对积压、丢包、异常进行观测和日志记录

### 风险 5：翻译拖慢整体实时性
应对：
- 翻译层后置
- 与 STT 主链路解耦
- partial 翻译降频，final 优先

### 风险 6：Windows 打包与音频依赖兼容性问题
应对：
- 在 GUI 阶段前就开始做最小验证
- 不把依赖兼容问题拖到最后集中处理

---

## 10. 执行建议

推荐顺序：
1. 先完成 M0，把不确定性压到最低。
2. 再完成 M1 和 M2，尽快打通“采音 -> 转写”。
3. 接着完成 M3，拿到可用的 MVP 主链路。
4. 用 M4 做稳定性加固和 MVP 验收。
5. MVP 通过后，再进入 M5、M6、M7。

决策规则：
- 任何不直接影响 MVP 成败的细节，优先后移，不在主文档写死。
- 任何会影响架构边界和验收标准的内容，应明确写入主文档。
- 遇到实现细节分歧时，优先选择更简单、可观测、可回退的方案。
